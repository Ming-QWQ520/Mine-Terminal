1. Mine - Terminal 项目概述

开发一个适用于 Minecraft Java 版 1.20.1、Forge 47.4.20 的客户端模组。
核心目标是：在游戏内提供一个完整的伪终端（PTY）驱动、支持 ANSI/XTerm 转义序列的全功能系统终端。玩家可以在 Minecraft 窗口中直接运行包括 vim、tmux、htop 等在内的全屏终端程序，体验与操作系统原生终端模拟器毫无差别的交互能力。

2. 背景与动机

· Minecraft 模组运行在 JVM 上，Java 原生通过 ProcessBuilder 只能创建标准管道，缺少伪终端（PTY），导致许多交互式全屏程序无法工作或异常。
· 通过引入 PTY 库（如 pty4j）与终端模拟渲染库（如 jediterm），可以在 Java 中实现一个功能完备的终端模拟器，并嵌入 Minecraft 的 GUI 系统。
· 本模组致力于抹平虚拟游戏与真实操作系统之间的最后一道鸿沟，为高级用户、开发者、系统管理员提供在游戏内直接操作完整 Linux/Windows 终端的能力，也可用于教学、极客整合包等场景。
· 必须在显著位置告知用户：本模组会赋予游戏完整的操作系统权限，使用者需自行承担相应安全风险。

3. 核心功能需求

3.1 伪终端（PTY）支持

· 使用 PTY 库创建伪终端，使运行在其中的进程认为自己连接到了一个真实终端，从而能够获取终端窗口大小、支持信号、支持作业控制等。
· 支持动态调整终端窗口大小：当玩家改变终端 GUI 尺寸时，模组必须向 PTY 发送 TIOCSWINSZ 信号（或等效操作），更新终端的行数与列数，以保证全屏程序布局正确。

3.2 完整 ANSI/XTerm 转义序列解析

· 集成终端模拟库（如 jediterm 的终端内核），解析程序输出的 ANSI 转义序列，包括但不限于：
  · 光标移动（上下左右、绝对定位）
  · 字符属性（颜色、加粗、下划线、闪烁、反色等）
  · 屏幕清除（清屏、清行）
  · 滚动区域控制
  · 鼠标事件（可选支持 xterm mouse tracking）
  · 备用屏幕缓冲区（例如 vim 使用的 altscreen）
· 所有转义序列的视觉效果须在游戏 GUI 中准确呈现，与标准终端模拟器（如 iTerm2、Windows Terminal）表现一致。

3.3 终端 GUI

· 提供一个可全屏或窗口化显示的终端界面，背景半透明或不透明可配置。
· 输出区域基于终端模拟库的渲染输出，使用等宽字体绘制字符网格。
· 支持自定义配色方案（至少提供暗色/亮色两套预设），以及字体大小、行间距、光标样式（块状、下划线、竖线）等视觉元素。
· 光标闪烁效果需正确渲染。
· 支持鼠标交互：鼠标点击位置应能转换为终端字符坐标，并发送给 PTY（需终端程序开启鼠标支持）。
· 支持鼠标滚轮滚动终端的回滚缓冲区（终端模拟库通常会维护回滚历史）。

3.4 键盘输入处理

· 所有玩家键盘输入必须直接传递给终端模拟库的输入接口，由其转换为 PTY 写入，而不是简单写入 stdin。这确保特殊键（功能键、方向键、Ctrl 组合等）能够产生正确的转义序列。
· 必须正确处理以下按键映射：
  · 可打印字符、Enter、Backspace、Tab、Escape
  · 方向键（上下左右）→ 生成 ESC 序列
  · Home、End、PageUp、PageDown
  · 功能键 F1-F12
  · 组合键：Ctrl+A..Z、Ctrl+[ 等（发送对应控制字符）
  · 现代终端常用组合：Ctrl+Shift+C/V（复制粘贴，需与系统剪贴板交互）
· 需要将 Minecraft 的键盘事件（keyPressed、charTyped）转换为终端模拟库可识别的按键/字符，并处理因键盘布局差异导致的字符映射问题。

3.5 系统 Shell 启动

· 支持自动检测操作系统，但 PTY 创建方式可能因平台而异：
  · Linux/macOS：通过 PTY 启动用户配置的 Shell（如 /bin/bash、/bin/zsh），继承系统环境变量，但可选择性清除或设置 TERM 变量为 xterm-256color 以保证功能完整性。
  · Windows：需使用第三方解决方案（如 WinPTY 或 ConPTY），或要求用户使用 WSL、Cygwin 等环境。需明确文档说明 Windows 下的限制与配置方式。
· 允许用户通过模组配置文件指定 Shell 路径、启动参数、初始工作目录等。

3.6 打开、关闭与管理

· 默认快捷键打开（可配置，建议默认 Ctrl+Shift+T），也支持聊天命令 /terminal 开启。
· 按 Esc 需由终端模拟库内部处理（如 vim 中 Esc 是功能键），只有当终端界面处于“非程序捕获”模式时，Esc 才用于关闭终端界面。可引入“终端焦点”概念：点击终端区域获取焦点，此时所有按键发送给终端；按特殊组合键（如 Ctrl+Shift+Q）强制退出终端界面并终止进程。
· 关闭界面时，确保向 PTY 进程发送 SIGHUP（或调用 destroyForcibly），并释放所有相关资源，防止僵尸进程。

3.7 多会话支持（扩展需求）

· 允许在同一游戏客户端内开启多个终端标签页，每个标签页对应独立的 PTY 进程。
· 标签页可命名、可关闭，界面提供标签栏。

4. 技术实现要求

4.1 Mod 加载器与 JDK

· Forge 1.20.1 - 47.4.20，使用 Forge MDK。
· JDK 17。
· 必须使用 Gradle 引入外部依赖：pty4j（提供 PTY 支持）与 jediterm-core（提供终端模拟与渲染）。

4.2 PTY 集成

· 使用 pty4j 的 PtyProcessBuilder 创建 PTY 进程，配置终端尺寸、环境变量、工作目录等。
· 获取 PtyProcess 后，分别处理输入输出：
  · 输出流：通过终端模拟库的 TtyConnector 接口连接到 jediterm 的 JediTerm 后端。
  · 输入流：当终端模拟库产生输入数据时，写入 PtyProcess.getOutputStream()。
· 监听 PTY 进程退出事件，在终端界面上显示“进程已结束”提示，并允许重启。

4.3 终端模拟渲染

· 利用 jediterm 的核心，创建一个终端后端（如 TtyBasedArrayDataStream），将 PTY 的输出馈入。
· jediterm 会将解析后的字符、样式、光标位置存储在一个二维字符网格中。在 Minecraft 的 Screen.render 方法中，将这块网格绘制到 GuiGraphics 上：
  · 根据终端字体大小和行/列数计算绘制区域。
  · 遍历每个字符格，从网格获取字符、前景色、背景色，使用 Minecraft 的 Font 绘制（若需保证等宽和特殊字符，可能需要引入一个专用的等宽字体纹理，或通过 TTF 渲染，但为简化可先使用 Minecraft 默认等宽字体，并正确处理 Unicode）。
· 实现光标渲染：根据终端模拟库提供的光标位置和样式，绘制闪烁光标。
· 处理终端的回滚缓冲区：jediterm 内部管理历史行，GUI 可通过鼠标滚轮查看，并自动跟踪或手动锁定。

4.4 键盘转换层

· 编写一个适配器，将 Minecraft 的 GLFW 键盘事件（key code、scan code、modifiers）以及 charTyped 的字符转换为 jediterm 接受的 KeyEvent（或字符流）。
· 特别注意：
  · 处理系统键盘布局：通过 GLFW 的 charTyped 获取实际输入字符。
  · 非字符按键（方向键、功能键）需要使用 jediterm 预定义的转义序列，可通过其 TerminalWriter 直接写入。
· 在终端程序捕获鼠标事件时，需将鼠标坐标和点击信息编码为 xterm mouse sequences 并通过 PTY 输入。

4.5 窗口尺寸同步

· 当 Minecraft 调整窗口大小、终端 GUI 改变尺寸时，重新计算终端行/列数（基于字体尺寸），并调用 PTY 的 setWindowSize(columns, rows) 方法，同时通知终端模拟库调整终端尺寸。
· jediterm 支持动态调整尺寸，会触发重绘。

4.6 配置文件与自定义

· 配置文件 config/fullterminal-client.toml，包含：
  · 终端 Shell 路径与参数
  · 默认终端尺寸（行列或“从窗口自动计算”）
  · 字体：可选择“游戏默认等宽”或指定外部字体文件路径（进阶）
  · 配色方案：预定义主题或自定义前景/背景/ANSI 16色
  · 快捷键绑定
  · 回滚行数限制
  · 鼠标事件模式（禁用/总是/仅当程序请求）
  · Windows 下的特殊设置（如使用 ConPTY 路径）

4.7 跨平台考虑

· Linux/macOS：pty4j 直接使用系统调用 fork/exec，原生支持。
· Windows：pty4j 在 Windows 上使用 WinPTY（ConPTY）作为后端，需要分发额外的原生库（winpty-agent.exe 等），或者要求用户系统版本（Windows 10 1809+）支持 ConPTY 并通过 com.pty4j.windows.WinPty 配置。在文档中详细说明 Windows 部署步骤。

5. 安全与风险控制

· 模组发布时必须显著声明其功能及潜在风险：可执行任何系统命令、修改文件、访问网络等，相当于将操作系统最高权限暴露在 Minecraft 进程中。
· 模组设计为客户端专用，在服务端加载时必须自禁用并输出错误信息。
· 不应包含任何网络监听功能，避免远程恶意访问。
· 建议用户不要以管理员/root 身份运行 Minecraft。
· 代码中必须避免命令注入漏洞（尽管输入来自本地玩家，仍需使用安全 API）。
· 关闭终端时需确保子进程被终止，并清理临时资源。
· 若将来添加“脚本触发”功能，必须加入严格的白名单和确认机制。

6. 非功能需求

· 性能：终端渲染应高效，使用脏矩形优化或只在有数据更新时刷新对应行。避免每帧全量绘制字符网格造成卡顿。
· 稳定性：PTY 进程意外退出不应导致 Minecraft 崩溃；终端模拟库异常时应能捕获并安全回退。
· 兼容性：明确支持的平台（Windows 10+、macOS 10.14+、主流 Linux 发行版），并声明不可用的环境（如 headless 服务器无 GUI）。
· 本地化：界面提示文字（状态栏、关闭按钮等）需支持国际化，至少提供英文和中文。

7. 潜在扩展

· SSH 客户端集成：直接在终端内运行 ssh，通过 PTY 无缝使用。
· 终端录制：以 typescript 格式保存会话记录。
· 与 Minecraft 事件联动：允许红石信号等事件触发白名单命令，谨慎开放。
· 自定义配色与字体：支持导入 iTerm2 配色方案，或加载 TTF/OTF 字体。

8. 开发阶段建议

1. 搭建 Forge MDK，引入 pty4j 和 jediterm 依赖，验证 PTY 在目标平台的可工作性。
2. 在独立测试中完成终端模拟库与 PTY 的对接，确认普通命令和全屏程序（如 vim、htop）运行正常。
3. 创建一个基本的 Minecraft Screen，将终端模拟库的字符网格绘制到屏幕上。
4. 实现键盘事件到终端输入的转换，处理特殊键和组合键。
5. 实现窗口大小变化同步。
6. 加入多标签、配置系统、快捷键和界面美化。
7. 全面跨平台测试（Windows ConPTY/WSL、Linux原生、macOS），解决编码和字体问题。
8. 编写详细的使用说明、安全警告、配置文件文档，准备分发。

9. 注意事项

· 依赖的原生库（pty4j 包含平台相关的二进制文件）必须正确打包进模组 JAR。
· Minecraft 的字体渲染可能不支持部分 Unicode 字符（如 powerline 符号），需说明或提供替换方案。
· Windows 下的终端体验可能受限于 ConPTY 的成熟度，建议提供 WSL 作为推荐后端。
· 该模组的功能极为强大，发布时务必遵循平台规则，并明确其权限范围。
